# 2026-04-08 (later) — founding-breach migration: claude → hulk merged carrier home

A second long pass on the same calendar day, on a completely different
workstream. While the substrate normalization sweep above was reshaping

host-RAM crash from 2026-04-04/05 that gave hulk its name) and turned
it into a real on-disk merger: the legacy `harnesses/claude/` home
went away, and `harnesses/hulk/` became the canonical merged carrier
home. All of it happened *under live operation* — the running Claude
Code session never lost a tool call, never dropped a transcript line,
never panicked.

## Why it had to be live

The shape of the problem: the running Claude Code process held cached
absolute paths under `harnesses/claude/`. Naive moves break:

- `mv claude/* hulk/` — open file descriptors follow inodes (✅), but
  new file creation continues to land at `claude/` because the process
  re-`open()`s by cached absolute paths (❌).
- Re-point the home symlink → helps for things resolving live, but
  cached `realpath()` in the running process doesn't see it (❌).
- `cp -a` then delete → snapshots a moment in time, drops every write
  that lands in the source after the cp (❌).
- Compat symlink `claude → hulk` → operator-vetoed: "i don't like
  compat links because YOU forget about them" (❌).
- Bind mount / firmlink → not available mid-session on macOS (❌).

The technique that worked: **two paths, one inode.** Hardlink every
existing file in `claude/` to its sibling at `hulk/` so both paths
share the underlying inode, then run a small daemon that hardlinks any
newly-created files within the FSEvents coalescing window. Claude
Code keeps writing through the legacy path string; the data lives at
the canonical home; no compat shim exists at any layer.

## The hard finish

Late in the session the operator said "we should do a hard finish
(hah) of this before we sleep." The remaining `harnesses/claude/`
directory still existed on disk because the migration plan had been
to defer the final cleanup to a fresh session. A diagnostic check
revealed something better: **Claude Code was writing via `~/.claude/X`
paths and resolving the symlink at write time**, not via cached
absolute paths. The `claude/` directory had only the few files
manually hardlinked there earlier in the session — every new file
since the symlink swap had been landing directly at hulk. Five files
in claude/, 2832 in hulk/, all five hardlinked to their hulk twins,
zero unsafe, zero claude-only.

The hard finish itself:

```
pkill -f swift-hardlink-drain-cli  # drain daemon retired
rm -rf harnesses/claude/           # 14 MB of legacy entries
```

`hulk/` file count before the rm: 2832. After: 2832. Zero data loss.
The inodes were preserved by the hulk-side directory entries; only
the redundant claude-side directory entries went away. The session
kept running. The header kept rendering. The conversation kept
growing in the live JSONL inside hulk's tree.
