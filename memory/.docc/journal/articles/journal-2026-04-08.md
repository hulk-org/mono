## Why it had to be live

The shape of the problem: the running Claude Code process held cached
absolute paths under `harnesses/claude/`. Naive moves break:

- `mv claude/* hulk/` â€” open file descriptors follow inodes (âś…), but
  new file creation continues to land at `claude/` because the process
  re-`open()`s by cached absolute paths (âťŚ).
- Re-point the home symlink â†’ helps for things resolving live, but
  cached `realpath()` in the running process doesn't see it (âťŚ).
- `cp -a` then delete â†’ snapshots a moment in time, drops every write
  that lands in the source after the cp (âťŚ).
- Compat symlink `claude â†’ hulk` â†’ operator-vetoed: "i don't like
  compat links because YOU forget about them" (âťŚ).
- Bind mount / firmlink â†’ not available mid-session on macOS (âťŚ).

The technique that worked: **two paths, one inode.** Hardlink every
existing file in `claude/` to its sibling at `hulk/` so both paths
share the underlying inode, then run a small daemon that hardlinks any
newly-created files within the FSEvents coalescing window. Claude
Code keeps writing through the legacy path string; the data lives at
the canonical home; no compat shim exists at any layer.

the redundant claude-side directory entries went away. The session

### Lessons and decisions

- **Patch safety workflow formalized** and saved as
  `feedback_patch-safety-workflow.md`. Multi-chunk deletions or sweeping
  renames must grep-verify (symbols removed vs preserved) after each
  chunk, `git diff --stat` before every commit, and â€” for sustained
  refactors â€” scaffold a Swift Testing target with invariant tests
  early so each commit is measured against expected shape. Rismay
  flagged the risk explicitly after the ~2,000-line clia-mem
  deletions had zero test coverage behind them for most of the
  session.
- **Source Control (wrkstrm Inc) vs clia-mem (clia family)**: same
  codebase at the start, but product family and legal entity are
  durable decisions that have to be right before launch. Saved as
  `project_source-control-brand.md` and reinforced by the existing
  `user_legal-entities.md` rule.
- **clia-mem is single-lane by design** (Sessions only). The
  `CliaMemRoute` enum is kept around so future harness lanes
  (codex vs claude vs hulk vs claw sessions, etc.) can be added
  without having to rebuild the shell.
- **Review-before-apply via `.patch` files**, not an in-tool
  EditQueue. The Edit tool applies to disk immediately and there is
  no queue primitive; the pragmatic substitute is to author a patch
  file with `swift-git-cli repo patch create`, show it for review,
  and `repo patch apply` on approval. Dogfood on the Apply queue
  feature surfaced the pathspec scoping gap before I tried it on
  anything more load-bearing.
