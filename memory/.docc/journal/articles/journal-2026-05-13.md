## The two real bugs

1. **pi's `composition: "harness"`** had been there since pi was authored —
   silent because nothing audits an organism file unless explicitly asked.
   The enum has only been `{singular, collective}` at every version of
   `organism-schemas`. The value was never valid; it was just never read.
2. **swift-agent-cli was unbuildable for ~6 weeks** because
   `IncidentsImprintCommand.swift:147` referenced `receipt.charterRef` and
   `DoctorCommand.swift:143` referenced the retired `Charter` type. The
   `Charter` entity was dropped from `core-entity-set-v000-009-000` when the
   v1.0 conversation started naming `mission` / `operating-protocol` /
   `scenario` as the replacement vocabulary. Two callers were left
   referencing the dead name. One `charterRef` → `missionRef` rewrite and
   one case removal restored the substrate's primary audit tool.

## What "v1.0.0" actually means

The freshness rule: v1.0.0 takes the *latest of all current schemas* per
family at audit time. The `core-entity-set-v1.0.0.json` snapshot at
`schemas-sets/core-entities/v1.0.0/` is a freshness audit cut, not a frozen
binding — it moves as families publish new lines. As of today the
fitted values are:

```
schemaVersion (top-level)            = 0.5.0   (organism-schemas latest)
aspects.agent.doc.schemaVersion      = 0.4.0   (agent-schemas latest)
aspects.harness.doc.schemaVersion    = 0.1.0   (harness-schemas latest)
identity.schemaVersion               = 0.4.0   (identity-schemas latest)
composition                          ∈ {singular, collective}
```

## The LDT proof pair as audit convention

Every v1.0.0 fit lands two `.swift` files under `.tmp/repl-proofs/`:

- A **positive** proof that loads the organism JSON as a dictionary, asserts
  the schemaVersions match the v1.0.0 latest-per-family map, asserts the
  enum-valued fields use real values (composition, runtimeKinds), asserts
  identity coherence (id == identity.slug == aspects.agent.doc.ref.slug
  when an agent aspect exists), and prints `ok: <slug> organism fits v1.0.0`.
- A **negative** proof that synthesizes the pre-fit shape as a literal
  dictionary in Swift and confirms it fails *exactly* the predicates the fit
  corrected — by asserting `failures == expected`. The negative is what
  proves the positive is not vacuous.

Both run with `swift <file>`. The directory listing + git log of
`.tmp/repl-proofs/` is the audit index. The pattern matches the existing
substrate LDT discipline ("every non-trivial structural claim becomes a
compilable .swift proof").

## Afternoon — digikoma-recap landed (and what didn't work)

After the harness audit, scoped and built `digikoma-recap` at
`collectives/digikoma-org/.../intelligence/digikoma-recap/`. The mission:
hand it a chat, get back a structured `RecapModel`. Pure summarizer — no
chronicle writes, no bead spawning. Sibling to `digikoma-winddown` (which
is still spec-only); winddown is the eventual persistence layer on top.

### What's in the koma

Nine files following the digikoma convention: `アイディ.md`, `インスト.md`,
`レイ.md`, `スペック.json`, `lineage.json`, `Package.swift`, the Tool
sources with nine `@Generable` types (`DigikomaRecapModel`, `Arc`, `Win`,
`Decision`, `Bug`, `Rule`, `Followup`, `Caveat`, `Result` envelope), a
CLI executable, and three Swift Testing cases.

The engine wiring is one line:

```swift
let session = LanguageModelSession()
let response = try await session.respond(
  to: prompt,
  generating: DigikomaRecapModel.self
)
return response.content
```

That gives back a typed `DigikomaRecapModel` directly — no JSON parsing
on the koma side, no prompt-engineering-for-JSON. `@Generable` + `@Guide`
on every field do all the schema work.

### Build green, tests pass

`swift build` complete first try at 24.86 seconds. Three unit tests
covering identity stability, empty-transcript-throws, and default
argument values — all pass.

### Then I proved it against a real fixture and the bugs surfaced

Staged `.tmp/recap-fixtures/sample-session.txt` (~1.7 KB) with three
known bugs, three known blocked items, and one known durable rule.
Ran `digikoma-recap --input <fixture>`, captured the output.

Four real quality issues in the actual RecapModel JSON (not scaffold
issues — *output* issues):

1. **Backtick truncation.** Source fields containing backticked
   transcript content cut off mid-string. `bugsCaught[0].description`
   ended at literal `"Pi's \`composition: "` — truncated.
   `arcs[1].narrative` ended at `"…missing \`identity.profileType: "`.
   Diagnostic: `@Generable` String boundary breaks when source content
   contains backticks. Fix candidates: pre-strip / escape backticks
   before the FM call, or carry verbatim quotes in a separate
   `quotes: [String]` companion field.

2. **List under-extraction.** Fixture explicitly listed three bugs;
   recap returned one in `bugsCaught`. Three blocked items in fixture;
   one in `caveats`. The `@Guide` strings on list fields described
   shape ("array of bugs caught") but not extraction policy. Need
   "include EVERY X mentioned; do not condense or pick a representative."

3. **Hallucinated decision.** `decisions[0]` claimed *"How should we
   handle Pi's organism? → Design an engine-binding shape"* — the
   fixture discussed engine binding for **apple-pi**, not pi. The FM
   conflated them. Worse, the `rationale` said *"The validator did
   not catch Pi's bug"* — fixture said the opposite. This is the
   dangerous failure mode: hallucinated decisions, if consumed by
   winddown's bead-spawning, would produce phantom work items.

4. **Non-deterministic between runs.** Same fixture, two runs, two
   different titles and arc structures. Expected for FM but means
   regression tests need tolerance bands (count ranges, key-phrase
   presence) rather than exact-match.

### Verdict downgrade

Earlier in the conversation I called the koma 5/10 based on build
tests. After the proving run, it's closer to 3/10 — drops 60-70% of
grounded content and invents at least one decision. The scaffold is
right; the *behavior* needs grounding work before it can be trusted.

### Open follow-ups (beads spun)

1. `digikoma-recap-strip-backticks-preprocess` — fix the truncation
2. `digikoma-recap-tighten-list-guides` — fix the under-extraction
3. `digikoma-recap-add-grounding-post-validator` — fix the
   hallucination by asserting key phrases appear in transcript
4. `digikoma-recap-stage-regression-fixture` — promote the fixture
   into `digikoma-recap/tests/fixtures/` with tolerance-band assertions

The integration bead (`wire-digikoma-recap-into-digikoma-winddown`) was
spun earlier in the session — it covers the winddown side.

### What I learned

- **Build green ≠ it works.** I called the koma a respectable scaffold
  on the strength of compile + 3 unit tests. None of those tests covered
  the load-bearing FM-call path. The audit-not-eyeball-clone rule we
  saved this morning applies here too: validate the actual *behavior*,
  not just the structural fit.
- **@Generable is excellent for structure, weak on extraction policy.**
  The schema constrains the *shape* of the output rigidly. It does
  *not* enforce coverage ("get every bug"), grounding ("don't invent"),
  or completeness. Those have to come from prompt + post-validation.
- **One real fixture run is worth more than ten unit tests** for komas
  whose primary work happens inside an FM call.

### Two memories, one lesson each

**`feedback_no-owl-suffix-on-komas.md`** — never name a Koma with the
`-owl` suffix. Operator correction during the secret-labs design: "ajnd
please stop with the owls. we are not harry potter." Pattern crept in via
existing `privacy-review-owl` and chronicle references to
`launch-gate-owl + design-token-owl`. Going forward: plain slugs
(`secret-labs`, `privacy-review`, `baseline-capability`,
`architecture-review`, `code-review`).

**`feedback_human-visual-verification-before-user-visible-actions.md`** —
the vapor-wares.com near-miss in another chat where Hello World's product
page was about to ship as the Vapor Wares store front because every
automated gate had passed. Operator caught it via "I have not even seen
the site!" Codified into the schema as `HumanVisualVerification`
(verdict + notes + screenshot path) + `LaunchGate.requiresHumanVisualVerification`
computed marker on `.appStoreConnect` / `.githubPagesPublished` /
`.releaseArtifactHosted` / `.stagedRolloutPlan` / `.goNoGo` /
`.humanVisualReviewPassed`. Quote pinned: "this is WHY needing a human to
verify is important still. I hate human in the loop, but goddamn. human in
the loop."

### One pivot worth remembering

Almost rebuilt `desktop-studio` from scratch. Operator stopped me with
"don't we already have these things? read the code please." Located the
existing 10,365-LOC prototype at
`wrkstrm-app/private/apple/prototypes/desktop-studio/` with its
`DesktopStudioPlugin` model (5 plugins registered, plugin-kind enum, host
enum, origin enum, status enum, surface layer / place primitives, option
pane factories, canvas / inspector / workspace / onboarding views, catalog
tests). Pivoted from "build a new app" to "add a schema family that the
existing prototype can adopt": `desktop-studio-plugin-schemas` gives JSON
descriptors with bundle-ID identity + sidecar known-IDs, so the
prototype's static enum-based catalog can later be refactored to a JSON
loader without recompiling the host every time a new app wants to register
a plugin. Lesson: "read the code please" is a real instruction — running
grep first would have saved me hours of duplicate work.
