## How the session started

`/sync >h:hulk >c:wrkstrm-components`. The operator wanted to commission the
wrkstrm-components collective properly — give it ikigai, bring it to v1.0.0,
make it not look like an abandoned scaffold.

## How the session escalated

The first surprise was that wrkstrm-components didn't have ikigai because
**no one in the substrate had ikigai**. `core-entity-set-v1.0.0` sat on disk
as a release-candidate SPEC — fully built, fully tested by its own
`OrganismRoundTripTests` — but zero agents were bound to it. The cadence bead
`identity-schema-set-refs-sweep` was BLOCKED waiting on operator
authorization to sweep agents from `v0.7/v0.8` to `v0.9`. The v1.0.0 line
was further out, unbound, sitting like a tin can no one had opened.

So we became the canaries. wrkstrm-components and wrkstrm got commissioned
as the first v1.0.0 organisms. Then 5 operators (jakor, johnwhitecastle,
khegh, tkoh, uptobat) followed. Six commissioned bundles in one session.

## The harvest correction

I tried to be safe and follow the 2026-03-15 harvest pattern from the
operators' canonical homes: copy the shadow into a
`harvest/2026-05-11/wrkstrm-identity-shadow/` subdirectory, write an audit
markdown, then delete the shadow. The operator's reply:

> we weren't supposed to harvest. we were supposed to move so we can stop
> doing "harvest" directories

Lesson: the harvest pattern is retiring. Direct moves are preferred — fewer
strays, fewer audit files, less bureaucratic ceremony for cleanup that's
obvious in the diff. I undid the harvest, moved the files directly into
each operator's `identity/`, and trashed the audit markdowns.

## The roles-vs-agents correction

I had classified ios-engineer, swift-architect, product-manager,
project-manager as "agent members" of wrkstrm. The operator corrected me:

> these are roles ^^^
> and we were supposed to delete those from agents and move them to
> wrkstrm roles.

So roles are not members. Roles are reusable mandates that get filled by
members. The 4 misfiled identity bundles got harvested into proper
`OrgRoleModel` files at `wrkstrm/roles/[slug].org-role.json`, closing the
broken `activeRoleRefs` references in the workstream block that had been
sitting unresolved since the workstream pattern was introduced.

## The openclaw-discord investigation that wasn't

The 5 operator shadows in `collectives/wrkstrm/private/universal/identity/`
carried restricted-connector content — `tools: deny-all`, `x-onboarding:
^clip`, "Do not run git commands without explicit operator approval." It
looked like an openclaw or Discord bot was misrouting external-connector
profiles into the wrong path.

The git archaeology said no.

`git log --all --diff-filter=A` on the shadow files pointed at exactly one
commit: `e7e34f2a`, dated 2026-03-23 05:12:16 -0700, authored by Cristian A
Monterroza, message "Migrate identities from 0.5.0 to 0.6.0". The shadows
were the operator's own migration artifacts — placed correctly under the
0.6.0 doctrine at the time, then abandoned when the substrate moved
operators to `substrate/operators/[slug]/`. Their content (the restricted
onboarding profile) predates the migration by 5 weeks (Feb 17 timestamps
in the JSON body), so the migration just carried existing restricted-
connector content forward into a path that didn't make sense in the new
doctrine.

The "openclaw discord thing" was a reasonable hypothesis from the content,
but timing told a different story. The 2026-03-23 incident on disk —
`2026-03-23-oss-adoption-blocked-by-startup-and-organism-drift` — started
the same day as the migration commit, which is consistent with this being
the same doctrine-drift event, not two separate ones.

The cleanup is a one-shot fix. No connector to source-fix. The lesson for
future schema-set migrations is: include a "retire stale duplicates" step
when canonical home semantics change between sets.

## The sanity migrators

After the commissions landed, the operator asked:

> can we get 0.9.0 to 1.0.0 migrators even though we won't really need them?
> just as a sanity thing.

The substrate already had three migrators (`v0.6.0` to `v0.7.0`, `v0.7.0`
to `v0.8.0`, plus pre-existing fixtures). The chain ended at v0.8.0.
Building v0.9.0 to v1.0.0 was the next step. The operator then said "can
we go the 0.8.0 to 0.9.0?" and we filled that gap too.

Both migrators carry declarative `SchemaMigratorModel` descriptors that
enumerate preserves, synthesizes, drops, and warnings. The library targets
build green; 9 smoke tests pass; no CLI executables (no live consumers
need them).

The lesson: sanity migrators have real value even without live consumers.
They capture migration semantics declaratively in Swift, give future
migration runners a typed path forward, and surface contract decisions
that would otherwise require diffing two manifest JSONs side by side.

## The chronicle JSON repair

This very journal article is from the same session as a quiet meta-fix:
claude's chronicle file had a pre-existing JSON malformedness. The
2026-05-08 hello-world-google entry had been appended *outside* the
`entries` array, sitting as a sibling top-level object connected to the
main envelope by a stray `,`. `JSONSerialization` errored with "Garbage at
end around line 1211." I re-anchored that orphan into `entries[...]` and
added this commission/migrator entry properly, then deleted the trailing
orphan content. Now the chronicle validates: 25 entries, single
top-level object, clean close.

The shape of that bug is informative. Some automation (or some hand-edit)
appended the new entry by concatenation rather than insertion. Whatever
codepath produced that should be examined — but that's a separate
investigation, not this session's work.

## What I'd remember

- Operators belong outside collectives. The doctrine evolved; old
  placements stayed.
- Direct moves > harvest-with-audit. The audit ceremony is retiring.
- Roles are not members. Members fill roles.
- Sanity migrators are a real deliverable, not a stub.
- Git archaeology beats hypothesis when the dates are knowable.
- Small chunked Edits work; one giant Edit doesn't, because some chars
  in the new_string break the tool-call parser.
- The chronicle file format is fragile; appends need to land inside
  `entries[]`, not as siblings of the envelope.
