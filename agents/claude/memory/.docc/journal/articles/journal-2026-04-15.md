# Journal - 2026-04-15

## apple-pi plugin architecture: KomaAgentToolAdapter + AgentPlugin + ProcessPlugin

### What shipped

#### KomaAgentToolAdapter (ApplePiKomaBridge)

New `ApplePiKomaBridge` library target in `swift-apple-pi-cli`. Bridges the
two tool protocols: `Logikoma` (typed `Input`/`Output`) and `AgentTool`
(untyped `[String: AnyCodableValue]`).

Decode path: params dict -> JSON data -> `K.Input`. Encode path: `K.Output` ->
JSON string -> `AgentToolResult`. Factory `.asAgentTool()` on `Logikoma`.

```swift
KomaReadFileTool().asAgentTool(description: "Read a file from disk")
```

9 tests in `ApplePiKomaBridgeTests`. Only constraint: `K.Input` and `K.Output`
must be `Codable` - every koma in the catalog already is.

#### AgentPlugin + TurnContext

Replaced the ad-hoc `beforeTurn`/`afterResponse` closures on `AgentLoopConfig`
with a formal `AgentPlugin` struct. Plugins compose:

- `onTurnStart(_ context: TurnContext)` - fires once before the loop; results
  concatenate across plugins and are pinned to context for all LLM calls.
- `onTurnEnd(_ text: String, _ context: TurnContext)` - fires after the final
  assistant stop; transforms chain across plugins; only affects the `turnEnd`
  event (session JSONL stores raw response).

`TurnContext` carries `prompt`, `sessionFile`, `workingDirectory` - gives
plugins the information they need without coupling them to the loop internals.

13 tests updated + added in `AgentLoopHookTests`.

#### ProcessPlugin + ProcessPluginCommand

`ProcessPluginCommand: Sendable, Codable` - declarative JSON description of a
subprocess to invoke at a turn boundary. Template tokens in args:

| Token | Substituted with |
|---|---|
| `{{prompt}}` | User prompt for this session |
| `{{sessionFile}}` | Absolute path to the JSONL session file |
| `{{workingDirectory}}` | CWD at loop start |

Subprocess also receives env vars: `APPLE_PI_PROMPT`, `APPLE_PI_SESSION_FILE`,
`APPLE_PI_WORKING_DIRECTORY`, `APPLE_PI_REASON`.

`reason` field documents why the command runs - surfaced as `APPLE_PI_REASON`
for observability and audit.

```json
{
  "id": "turn-planner",
  "onTurnStart": {
    "executable": "koma-turn-planner",
    "args": ["--prompt", "{{prompt}}", "--session", "{{sessionFile}}"],
    "reason": "Generate a phased turn plan before the LLM responds"
  }
}
```

3 tests in `ProcessPluginTests` exercising `/bin/echo`, `/bin/cat`, and JSON
round-trip.

### Decisions

- `onTurnEnd` transforms only the event stream - session JSONL stores the raw
  response. The header is a display-layer concern, not session truth.
- `ProcessPluginCommand: Codable` so plugin config lives as JSON - no Swift
  compilation required to add or change a plugin.
- Two context channels (args tokens + env vars): args are for the subprocess's
  own argument parser; env vars reach library code that doesn't control its
  args.
