## What the day actually built

### `digikoma-recap` — the structured-output summarizer

Pure transcript-to-`RecapModel` koma. Nine `@Generable` typed structs
(`RecapModel`, `Arc`, `Win`, `Decision`, `Bug`, `Rule`, `Followup`,
`Caveat`, `Result` envelope). Engine wiring is a single line:
`session.respond(to: prompt, generating: DigikomaRecapModel.self)`.
No JSON parsing on the koma side; `@Generable` + `@Guide` do the schema
work.

Initial proving run against `.tmp/recap-fixtures/sample-session.txt`
surfaced four real quality bugs (not unit-test bugs — *output* bugs):
backtick truncation in string fields, list under-extraction (3 → 1),
hallucinated decisions, non-determinism between runs. Three were
fixable in this session:

1. **Backtick stripping** pre-processes the transcript before the FM
   call — replace ``` with `'''` and `` ` `` with `'`. The `@Generable`
   String boundary was breaking on backticked content.
2. **Tightened `@Guide` extraction policy** on every list field with
   "include EVERY X mentioned; do not condense" language. Re-run took
   `bugsCaught` from 1 → 3, `caveats` from 1 → 3.
3. **Grounding gate** (`groundRecap`) demotes items whose anchor words
   don't overlap with the transcript at all. Hallucinated items go to
   `caveats` with `[ungrounded]` prefix instead of corrupting the output.
4. **Non-determinism** is accommodated by tolerance bands in the
   fixture regression test, not "fought."

### Polarity grounding gate

The 2026-05-13 winddown smoke test had caught the FM rendering
*"apple-pi organism not yet authored"* as a win titled
*"Pi's Organism Authored"*. The grounding gate (word-overlap) passed
it because "Pi", "organism", "authored" all appeared in the transcript.
The fix:

- Tightened `@Guide` on `Win`/`Bug`/`Decision` for past-tense / present-state
- New static `isPolarityFlipped(item:transcript:)` scans an 80-char
  window around each anchor word in the transcript for negation/
  blocking markers (`not yet`, `pending`, `blocked`, `doesn't exist`,
  `will be`, `needs to`, `todo`, `still open/pending/blocked`, etc.).
  31-entry marker list.
- 5 new unit tests cover the gate in isolation + the integrated path.

The gate is a *floor*, not a ceiling — creative phrasing
("destined to fail", "consigned to the backlog") will slip through.
A future semantic gate is a separate bead, not this one.

## What I learned

- **The `@Generable` macros are excellent for structure, weak on
  extraction policy.** They constrain the *shape* of the output
  rigidly. They do not enforce coverage, grounding, or completeness.
  Those have to come from prompt + post-validation. Three gates ship
  with digikoma-recap now: tightened `@Guide` strings, anchor-word
  grounding, polarity-flip detection. Each catches a different class
  of failure.

- **Three carrier kinds emerged.** Session-hosting harnesses
  (pi/codex/gemini/hulk) carry conversation vocabulary. Factory
  carriers (digikoma) carry deployment vocabulary. Clide is a third
  kind: lifecycle-hosting (daemon membrane). The v0.2.0
  `sessionSchemaRefs`/`deploymentSchemaRefs` covers the first two
  cleanly; the third either fits under "session" loosely (Option B)
  or wants its own field (Option D, needing a v0.3 bump).

- **The freshness-rule test is the substrate's enforcement of
  "v1.0.0 = latest of all current schemas."** Without it a family
  bump that doesn't propagate to the v1.0.0 binding silently violates
  the doctrine. With it, the doctrine *fails the build*. This is the
  same pattern as LDT proof pairs — make a structural claim
  compilable.

- **Honest duplication beats silent uniqueness.** Both clia and
  claude chronicles now carry the same two stranded entries because
  both agents were active in the session. Picking one home would
  discard real signal.

- **One real fixture run is worth more than ten unit tests** for
  komas whose primary work happens inside an FM call. The smoke test
  surfaced four real quality issues that no @Test could have caught
  in isolation.

### Why this package exists

Yesterday's session forged the launch-gate proof grammar (six schema
families × ~155 tests × hello-world-google v1.0.0-b1's honest blocked
receipts). The grammar is data; humans need to SEE that data before they
can verify it. The schema-shaped `HumanVisualVerification` codifies the
verdict capture; the operator's reframe ("a human can click a button!
that's all we need") sent CLI productization back and pulled the work
toward a SwiftUI button + notification + artifact viewers. This is the
viewers piece.

### What landed

| `ArtifactViewerMode.swift` | Closed enum: `.compact` / `.full` / `.preview` — same data, three rendering registers |
| `PrivacyDisclosureReleaseView.swift` | App-Store-nutrition-label-shaped render: tracking flag → domains → collected types grouped by purpose → required-reason APIs → purpose strings → policy URLs |

Build green first try across 5 cross-package schema deps. 9/9 tests pass.

### Three lessons

- **Schema-first paid off in compile time**: the viewers consume Codable
  types pinned by ~155 schema tests; if the schemas decode their fixtures,
  the views can render them. Zero shape-mismatch errors at build because
  the schemas ARE the source of truth.
- **The `ArtifactViewerMode` register pattern collapses 3× UI surface**:
  same data renders as compact (one row), preview (card), full (detail)
  via one switch in `body`. No separate `ListItem` / `CardView` /
  `DetailView` class hierarchy to keep in sync — SwiftUI's view
  composition is the right abstraction.
- **`FallbackArtifactView` covers the long tail honestly**: any
  `ReleaseOperationArtifactKind` we haven't built a dedicated viewer for
  renders as a kind-badge + path pill. The release-verification-ui can
  dispatch `[ReleaseOperationArtifactKind: any View]` and fall back when
  the kind isn't registered. Means new artifact kinds light up
  immediately without UI work; UI work comes when the kind earns it.
