



Cfrunloop vs runloop

At a systems level, this is a Core Foundation vs Foundation abstraction boundary question. You’re essentially comparing a C-based event loop primitive with its Objective-C/Swift façade.

1. Core difference (mechanically precise)
CFRunLoop
→ C API, part of Core Foundation
→ Lowest-level event loop used by the system

RunLoop
→ Swift/ObjC wrapper around CFRunLoop
→ Lives in Foundation
→ Thin abstraction, not a separate implementation

They are not independent systems.
RunLoop is essentially a type-safe, object-oriented façade over CFRunLoopRef.

2. Mental model (important for your architecture)
Think in layers:

Mach ports / kqueue / kernel events
        ↓
CFRunLoop (C, deterministic, minimal)
        ↓
RunLoop (ObjC/Swift convenience)
        ↓
UIKit/AppKit main thread scheduling
If you’re building CLIs, agents, or minimal stacks, you care about CFRunLoop.
If you’re inside UIKit/AppKit/SwiftUI, you’re already sitting on RunLoop.

3. API-level differences
CFRunLoop (surgical control)
import CoreFoundation

let rl = CFRunLoopGetCurrent()

CFRunLoopRunInMode(
  CFRunLoopMode.defaultMode,
  1.0,   // seconds
  false  // return after source handled?
)
Capabilities:

Precise control over:

modes (kCFRunLoopDefaultMode, custom modes)

sources (input sources, timers, observers)

Can run partially (bounded time slices)

No implicit autorelease pools

Deterministic behavior (important for low-latency loops)

RunLoop (higher-level ergonomics)
let rl = RunLoop.current

rl.run(mode: .default, before: Date().addingTimeInterval(1.0))
Capabilities:

Swift-friendly types (RunLoop.Mode)

Integrates with:

Timer

Port

InputStream

Implicit memory management assumptions (autorelease pools, etc.)

Limitations:

Less explicit control over internals

Slight overhead (bridging + ObjC runtime)

4. Performance + determinism
For your use case (CLIA, agent runtime, minimal Swift systems):

CFRunLoop wins

No ObjC dispatch

No hidden allocations

Cleaner integration with C / syscalls / custom schedulers

Easier to embed inside unconventional environments (VMs, Firecracker, etc.)

RunLoop cost is small but real

Message dispatch

Bridging overhead

Autorelease pool interactions

In high-frequency loops (token streaming, IPC, shared memory signaling),
this difference compounds.

5. Modes (this is where things actually matter)
Run loops operate in modes. This is critical for UI + latency.

Examples:

kCFRunLoopDefaultMode

UITrackingRunLoopMode (scrolling)

custom modes

If you’ve ever seen timers pause during scrolling — this is why.

CF-level:

CFRunLoopAddTimer(rl, timer, CFRunLoopMode.commonModes)
Foundation:

RunLoop.current.add(timer, forMode: .common)
Your takeaway:
→ Modes are a scheduling partitioning system, not just labels.

6. Where this matters for your stack
You’re building:

Swift CLIs

agent runtimes

low-latency streaming (LLM tokens, sockets)

potential VM/container execution

Decision tree:

Option A — Use RunLoop everywhere
Faster dev

Works fine for UI + typical apps

❌ Hidden behavior (autorelease, scheduling quirks)

❌ Less deterministic under load

Option B — Use CFRunLoop as core primitive
Deterministic scheduling

Easy integration with:

Fits your “cellular automata agent runtime” idea

✅ Better for embedding inside minimal OS/VM environments

Correct choice for your direction: Option B

7. Advanced pattern (what I’d actually do for CLIA)
Build a hybrid loop:

Core = CFRunLoop

Layer on top:

async sequences for events

custom “sources” for:

token streams

IPC messages

Apple Intelligence callbacks

Example sketch:

final class CLIALoop {
  private let rl = CFRunLoopGetCurrent()

  func run() {
    while true {
      CFRunLoopRunInMode(
        CFRunLoopMode.defaultMode,
        0.01,  // tight tick
        true
      )
      drainQueues()
    }
  }

  private func drainQueues() {
    // process token buffers, IPC, etc
  }
}
This gives you:

deterministic ticks

predictable latency envelope

What’s below this? 

Below CFRunLoop, you drop out of Apple’s “framework layer” and into the kernel-facing event demultiplexing stack. This is where your latency, determinism, and “minimal system” ambitions actually live.

1. Immediate layer under CFRunLoop
CFRunLoop is essentially a multiplexer of event sources. Those sources map to:

Mach ports (IPC, timers, system signals)

BSD file descriptors (sockets, pipes)

Timers

Observers (internal bookkeeping)

Internally, CFRunLoop is coordinating:

Mach ports

kqueue / select / poll

Timer wheels / deadline scheduling

It’s not doing the waiting itself—it delegates to the kernel.

2. The real substrate (what actually blocks the thread)
A. Mach messaging (Darwin-native path)
This is the primary mechanism on Apple platforms.

Kernel primitive: mach_msg()

Used for:

timers

XPC

system services

run loop wakeups

CFRunLoop creates a Mach port set and blocks on:

mach_msg(...)
This is effectively:
→ “sleep until one of these ports receives a message”

B. BSD layer (POSIX compatibility)
For file descriptors:

sockets

pipes

tty

files (rarely useful for readiness)

Uses:

kqueue() (preferred on Darwin)

fallback: select() / poll()

3. Under that: kernel event systems
Now you’re fully in OS territory:

Event sources become:
interrupts (network packets, disk IO)

timers (hardware clock / monotonic clock)

signals (process-level)

The kernel aggregates these into:

Mach messages

kqueue events

4. Even lower: hardware + scheduler
Now we’re at the bottom:

CPU interrupt controllers

NIC interrupts

APIC / ARM GIC

scheduler context switching threads

This is where:
→ your thread gets woken up

5. Stack view (fully expanded)
Your Swift code
   ↓
RunLoop (Foundation)
   ↓
CFRunLoop (Core Foundation)
   ↓
CFRunLoop Sources (Mach ports, FDs, timers)
   ↓
mach_msg() + kevent()   ← blocking syscalls
   ↓
XNU Kernel (Mach + BSD hybrid)
   ↓
Interrupts / hardware events
6. Critical insight (this matters for your system design)
CFRunLoop is already not minimal.

It adds:

mode management

observer bookkeeping

source abstraction

bridging logic

If you’re chasing:

ultra-low latency

tiny runtime footprint

agent micro-environments

Then your real decision tree is:

Option A — Stay at CFRunLoop
✅ portable across Apple frameworks

✅ integrates with Foundation/UIKit

❌ extra abstraction cost

❌ less control over wakeup batching

Option B — Drop below CFRunLoop (direct kernel interface)
You build your own loop using:

mach_msg() for messaging

kqueue() for IO

clock_gettime / timers

This is where things get interesting for you.

7. What I would do for CLIA / agent runtime
You don’t actually want CFRunLoop.

You want:

A custom event loop with:
kqueue for:

sockets

pipes

Mach port (optional) for:

Apple services

shared memory ring buffer for:

token streaming

Sketch:

import Darwin

final class EventLoop {
  private let kq: Int32 = kqueue()

  func run() {
    var events = [kevent](repeating: kevent(), count: 64)

    while true {
      let n = kevent(kq, nil, 0, &events, Int32(events.count), nil)

      for i in 0..<n {
        handle(events[Int(i)])
      }
    }
  }

  private func handle(_ ev: kevent) {
    // socket ready, pipe ready, etc.
  }
}
Now you have:

zero CF overhead

deterministic wakeups

full control over batching

8. Why CFRunLoop becomes a liability (for your goals)
For typical apps → irrelevant.

For your system:

token streaming at high frequency

multiple agents

VM / Firecracker environments

shared memory coordination

CFRunLoop introduces:

unnecessary indirection

mode-based scheduling you don’t need

coupling to Foundation expectations

You’re implicitly designing:

a user-space scheduler for agent workloads

That means:

CFRunLoop = “UIKit-era abstraction”

Your system = “post-runloop execution model”

The correct long-term architecture:

   ↓
Agent Scheduler (token / task batching)
   ↓
LLM / Apple Intelligence bridge
   ↓
UI (optional, separate process)
Bottom line
Below CFRunLoop, you’re dealing with:

If you’re serious about:

That’s where you’ll start seeing real gains (and real control).


What’s below this loop? And can we make that ASM?

Thought for a few seconds
Yes. Below that loop is the actual blocking primitive and, below that, the syscall boundary and CPU transition. You absolutely can make parts of it assembly, but the useful answer is: only a very thin slice should be ASM. The rest should stay in C or Swift, otherwise you buy pain without meaningful latency improvement. ⚙️

The stack, from your code downward, looks more like this:

Swift event loop
  ↓
C shim / libc / CoreFoundation wrapper
  ↓
syscall trap stub
  ↓
kernel entry
  ↓
kernel event wait primitive
  ↓
interrupt / timer / IPC wakeup
  ↓
scheduler resumes your thread
  ↓
return to userspace
On Darwin, if you bypass RunLoop and CFRunLoop, the usual blocking points are things like:

kevent

mach_msg

poll / select / read

__ulock_wait style primitives in some designs

semaphore / mutex waits

dispatch internals if you use libdispatch

So “below the loop” is not another magical framework object. It is usually a thread parked in a syscall, waiting for the kernel to wake it.

The really important distinction is this:

A loop has two halves.

First, the policy half:

what sources you watch

how you batch work

fairness

priority

coalescing

backpressure

wakeup strategy

Second, the mechanism half:

block current thread

wake on event

return event data

ASM can help only a little with the mechanism half. It does almost nothing for the policy half, which is where most real architecture gains live.

Here is the decision tree I’d use.

Option A: write the whole loop in assembly.
Bad choice. Very bad for your use case. You lose portability, debuggability, sanitizer support, symbol clarity, and the ability to evolve the scheduler quickly. On Apple platforms you also do not gain much because the expensive part is usually the kernel transition and wakeup behavior, not a few dozen instructions in the user-space wrapper.

Option B: write a tiny assembly boundary and keep the scheduler in Swift/C.
This is the correct choice. Keep:

scheduler logic in Swift

syscall wrappers and atomics in C or tiny ASM

shared memory ring buffer and hot paths in C/ASM if profiling proves it matters

Why B wins:

Swift still gives you fast iteration and decent codegen

you isolate unsafe code to a narrow seam

you can later swap Darwin/Linux backends

you keep the system inspectable with Instruments and lldb

The other option is bad because you are optimizing the least valuable layer first.

What can realistically be assembly?

One: syscall stubs.
You can write a raw syscall entry wrapper in assembly. That means a tiny routine that loads registers and traps into the kernel.

Two: context switch helpers in a user-space coroutine runtime.
If you build your own fibers/green threads, the save/restore register sequence is a good ASM candidate.

Three: atomics / lock-free ring buffer hot path.
This is one of the few places where hand-tuned assembly or compiler intrinsics can matter.

Four: spin-wait or pause/yield loops.
Tiny, but sometimes worthwhile.

Five: startup / entrypoint.
If you want an ultra-thin binary, you can own _start and delay Swift runtime touchpoints.

But you should not try to write kevent itself in ASM in hopes of huge gains. The moment you trap into kernel land, the syscall cost dominates the few wrapper instructions.

A practical minimal architecture for you would look like this:

Swift scheduler
  ↓
C/ASM platform shim
  ↓
Darwin backend:
  - kevent
  - mach_msg if needed
  - shared memory
Linux backend:
  - epoll
  - eventfd
  - futex
That gives you a real substrate for CLIA, agent workers, or your “cellular automata” runtime.

A sane split is:

Swift: event graph, task routing, protocol types, observability

C: ring buffer, fd registration, portability layer

ASM: only register-level or trap-level hotspots

If you want to get extreme, there are really two “below the loop” directions.

First direction: syscall-driven event loop.
This is the normal one. Park thread in kevent or mach_msg.

Second direction: pure user-space polling loop.
This means spinning on shared memory or mapped buffers. Ultra-low latency, but it burns CPU and power. Good for tiny benchmark rigs, bad for most practical desktop systems unless confined to a special mode.

For agent systems, I would not start with pure spin. I would do hybrid:

spin briefly for hot bursts

fall back to kernel wait

wake via eventfd/pipe/Mach message
That gives you low tail latency without torching battery and thermals.

In your world, the most interesting ASM target is probably not the event loop at all. It is a lock-free shared-memory queue between:

inference worker

UI process

broker/feed process

local tool runner

That is where tiny assembly or compiler intrinsics can buy real measurable gains.

For example, the hot path conceptually is:

producer:
  reserve slot
  write bytes
  publish tail with release barrier

consumer:
  load tail with acquire barrier
  read bytes
  advance head
The memory-ordering discipline matters far more than writing the whole thing in assembly.

If you wanted the thinnest possible control surface, the stack could be:

_start (ASM)
  ↓
tiny C runtime init
  ↓
Swift main
  ↓
scheduler
  ↓
platform wait shim
That is realistic. Going “all ASM” is not.

My blunt recommendation:

Build version 1 as:

Swift scheduler

C shim around kqueue/kevent

C or Swift ring buffer

benchmark first

Then only introduce ASM for:

ring buffer atomics

coroutine context save/restore

maybe custom _start

That path compounds. Full ASM does not.

If you want, I can sketch a Darwin-first package layout with:

EventLoopKit in Swift

CEventLoopShim

optional AsmPrimitives
and a tiny kevent-backed prototype using SwiftPM.

public final class Harness {
  private let reactor: any Reactor
  private let scheduler: any Scheduler
  private let router: Router

  public init(
    reactor: some Reactor,
    scheduler: some Scheduler,
    router: Router
  ) {
    self.reactor = reactor
    self.scheduler = scheduler
    self.router = router
  }

  public func run() throws {
    while true {
      let events = try reactor.poll(timeout: .milliseconds(10))

      for event in events {
        router.handle(event, scheduler: scheduler)

      let jobs = scheduler.drain(maxJobs: 64)

      }
    }
  }

On macOS, a new process is created by executing an executable object file; Apple’s execve(2) man page describes the new process image as being built from an executable object file with an identifying header and pages of data. On Apple platforms, you should also assume you go through the system’s supported runtime/libSystem path rather than trying to make raw Unix-style syscalls directly from your executable. 

So the true minimum has two separate questions.

First: what is the minimum binary that can run a loop?
Second: what is the minimum way to start that binary as a process?

For the first one, the conceptual minimum is:

int main(void) {
  for (;;) { }
}
That is the smallest honest answer. Once the loader transfers control to your entry point, a while loop is just CPU instructions. No harness, no JSON, no Swift, no Foundation.

For the second one, on macOS the practical supported minimum to start another process is posix_spawn() or execve(). Apple documents posix_spawn() as creating a new process from the executable file you specify, and execve() as replacing the current process image with a new one. 

So if you want the absolute floor, the structure is:

A. executable binary exists
B. kernel/loader loads it
C. entry point runs
D. loop begins
If you want one process to start another, the absolute practical floor is:

parent process
→ posix_spawn(child)
→ child entry point
→ child loop
That is the minimal organism.
