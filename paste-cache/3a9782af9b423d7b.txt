⏺ That's a strong architectural insight. A ~Copyable generic widget enforces the physical truth: a desktop widget is a unique resource that
  owns an NSWindow — you can't duplicate it, only move it.

  Here's the shape I'd propose:

  // What every widget state must provide
  protocol WidgetStateProtocol: Observable, AnyObject, Sendable {
    @MainActor func reset()
  }

  // The placement + appearance config (shared across all widget types)
  struct WidgetAppearance {
    var placement: WidgetPlacement  // renamed from ClockPlacement
    var opacity: CGFloat
    var background: Color
  }

  // The generic widget — ~Copyable means unique window ownership
  struct DesktopWidget<State: WidgetStateProtocol>: ~Copyable {
    let state: State
    var appearance: WidgetAppearance
    private var window: NSWindow?  // owned exclusively

    init(state: consuming State, appearance: WidgetAppearance) { ... }

    mutating func show()   // creates/updates the NSWindow
    mutating func hide()   // tears down the NSWindow
    deinit { hide() }      // ~Copyable gets deinit on structs
  }

  ~Copyable gives you struct deinit — when the widget goes out of scope, the window tears down automatically. No more manual hide()
cleanup.
  And consuming State on init means the state is moved into the widget, not shared — one state, one widget, one window.

  The clock becomes the first concrete widget:

  let clockWidget = DesktopWidget(
    state: InterviewClockState(),
    appearance: .init(placement: .overlay, opacity: 0.15, background: .clear)
  )

  Future widgets (system monitor, stock ticker, agent status) just provide a new State + View pair.

  struct DesktopWidget<State: WidgetStateProtocol, V: WidgetVisibility>: ~Copyable {

    var appearance: WidgetAppearance
    private var window: NSWindow?

  // hide() only exists when V == Shown
  extension DesktopWidget where V == Shown {
    consuming func hide() -> DesktopWidget<State, Hidden> {
      window?.orderOut(nil)
      return DesktopWidget<State, Hidden>(state: state, appearance: appearance)
    }
  }

  // show() only exists when V == Hidden
  extension DesktopWidget where V == Hidden {
    consuming func show() -> DesktopWidget<State, Shown> {
      // create window, configure placement...
      return DesktopWidget<State, Shown>(state: state, appearance: appearance, window: win)
    }
  }

⏺ This is sharp work. The ~Copyable + phantom type-state + consuming func trio is the right shape — you get compile-time enforcement that
  hide() only exists on a shown widget, and the consuming keyword means the old value is gone after the transition. No runtime checks.

  But there are two places the type-state promise leaks, and one place it can be made tighter.

  ★ Insight ─────────────────────────────────────

  1. The NSWindow? Optional is a runtime lie the type system is already trying to eliminate.

  In the Hidden state, window is guaranteed nil. In Shown, it's guaranteed non-nil. But the field is NSWindow? in both — the compiler can't
  see the invariant, and someone reading the struct has to manually reason about which phantom type implies which optionality.

  The fix: separate storage per state. Two ways to do it:

  // Option A — separate structs, shared protocol
  struct DesktopWidget<State: WidgetStateProtocol, V: WidgetVisibility>: ~Copyable {
    let state: State
    var appearance: WidgetAppearance
    // NO window field here — the extensions own it
  }

  // Shown state owns a non-optional window
  extension DesktopWidget where V == Shown {
    // window stored via a private associated wrapper or a nested container
  }

  But that's awkward in Swift because you can't conditionally add stored properties. The cleaner pattern:

  // Option B — enum-of-states, not phantom type
  struct DesktopWidget<State: WidgetStateProtocol>: ~Copyable {
    let state: State
    var appearance: WidgetAppearance

    enum Lifecycle: ~Copyable {
      case hidden
      case shown(NSWindow)  // non-optional, proven by construction
    }
    var lifecycle: Lifecycle
  }

  Then show() and hide() are mutating (not consuming) and pattern-match on lifecycle. You lose the "hide doesn't even exist in the type
  signature" compile-time guarantee, but you gain the "window is non-optional when shown" compile-time guarantee. Tradeoff worth naming.

  2. consuming State on a reference type is a paper promise.

  State: WidgetStateProtocol requires AnyObject (through Observable), so State is always a class. consuming on a class reference moves the
  reference, not the object. If the caller held another reference to the same InterviewClockState instance (and they will — SwiftUI views
  observe it), the "one state, one widget" ownership claim is unenforceable at compile time.

  That's probably fine in practice — the State is deliberately shared with SwiftUI for observation. But the init signature init(state:
  consuming State, ...) implies exclusive ownership it can't deliver. Consider dropping consuming from the init parameter and noting in the
  doc comment that state is shared-by-design with the view layer.

  3. The WidgetVisibility protocol can tighten to an enum with ~Copyable sealing.

  // Current — open to extension by anyone:
  protocol WidgetVisibility {}
  enum Shown: WidgetVisibility {}
  enum Hidden: WidgetVisibility {}

  // Tighter — sealed set, uninstantiable:
  enum Shown: ~Copyable {}   // no cases = can't instantiate
  enum Hidden: ~Copyable {}  // phantom-only

  Making them ~Copyable with no cases means they can only exist as phantom type parameters — nobody can accidentally construct a Shown()
  value or pass one around. They're pure compile-time tokens.

  ─────────────────────────────────────────────────

  The core insight — ~Copyable struct deinit for window lifecycle + type-state for show/hide — is correct and worth building on. The
  question is whether the phantom type-state (compile-time "hide doesn't exist on hidden widgets") or the enum lifecycle (compile-time
  "window is non-optional when shown") is the stronger guarantee for this use case. Both are real wins; they're just wins in different
  dimensions.
