Exactly. That triad is the right mental model 🔧

Passwords gives us secure access semantics.
Wallet gives us object importance and visual dignity.
Reminders gives us temporal pressure, recurrence, and “needs attention now.”

That combination is stronger than a plain accounts database because accounts are not static records. They decay. Tokens expire. Passwords rotate. MFA gets broken. API keys age out. Brokerage permissions lapse. KYC docs expire. Support PINs go stale. The app should treat account state as living operational inventory, not archival data.

So I’d tighten the product around three layers.

The first layer is the durable account object. That is the Wallet part. Each account card should feel like a first-class asset with institution, owner, masked identifier, status, risk, and linked access profile. It should be pleasant to scan and hard to confuse.

The second layer is the secure access graph. That is the Passwords part. An account can have one or more access profiles: login identity, API token, refresh token, passkey, MFA setup, recovery methods, support PIN, linked secret references. Importantly, the account is not the secret. The account owns a set of access capabilities.

The third layer is the operational timeline. That is the Reminders part. Every account should generate obligations and events:
token expires in 12 days,
password rotation due next week,
document renewal due in 45 days,
broker connection failed yesterday,
last verified login was 37 days ago,
2FA recovery not configured.

That means the app should not just store accounts. It should surface tension.

Right—what you’re circling is not one domain, it’s a composite system:

Identity → who

Auth / Access → how

Service → where

Most products collapse these. That’s why they degrade.

The clean move is to separate them structurally but unify them visually.

The correct mental model
Think in terms of a 3-axis graph, not a hierarchy:

Identity  ─── uses ───▶  Access  ─── grants ───▶  Service
   (who)                  (how)                  (where)
An Account is just a projection across those three axes.

That’s the key insight.

Why this matters (practically)
If you don’t separate these:

Identity leaks into credentials (bad for shared accounts)

Credentials get duplicated across services (API keys, emails, etc.)

Services become dumb labels instead of operational objects

You can’t reason about access health or expiration cleanly

You’ve already felt this friction.

Correct domain decomposition
1. Identity (who)
This is not just “user.”

It’s:

individual

joint identity

entity (LLC, trust, fund, etc.)

struct Identity: Identifiable, Codable {

  var kind: Kind
  var legalName: String
  var displayName: String
  var taxIDMasked: String?
  var residency: String?

  enum Kind: String, Codable {
    case individual
    case joint
    case entity
  }
}
2. Service (where)
The external system.

struct Service: Identifiable, Codable {
  let id: UUID
  var name: String        // "Schwab", "AWS"
  var category: Category
  var provider: String?
}

extension Service {
  enum Category: String, Codable {
    case brokerage
    case bank
    case cloud
    case saas
    case crypto
    case internal
  }
}
3. Access (how)
This is the most under-modeled piece in most systems.

Access is not just a password.

It’s:

login credentials

API keys

OAuth tokens

passkeys

support PINs

recovery mechanisms

  case login
  case apiKey
  case oauthToken
  case passkey
  case support
}
4. Account (projection layer)
This is where everything comes together.

struct Account: Identifiable, Codable {
  let id: UUID

  var serviceID: UUID
  var identityID: UUID
  var accessProfileIDs: [UUID]

  var status: Status
  var tags: [String]

  var openedAt: Date?

}
This is not the “root truth.”
It’s the operational view.

Can’t forget both security levels and or tokens… damn 

Right—and this is the part most apps completely miss.

You’re not just tracking “access.”
You’re tracking security posture over time 🔐

There are really two orthogonal dimensions you need to model explicitly:

1) Security Level (static posture)
This is how strong the access is.

Think of it as a computed score, not just a field.

Examples:

password only → weak

password + SMS → medium

TOTP / app MFA → strong

passkey / hardware key → very strong

API key with no scope restrictions → risky

API key with scoped permissions + rotation → strong

So instead of storing “MFA: true/false”, you want:

struct SecurityPosture {
  var authFactors: [AuthFactor]
  var recoveryConfigured: Bool
  var scopedAccess: Bool
  var lastAuditAt: Date?
}

enum AuthFactor {
  case password
  case sms
  case totp
  case passkey
  case hardwareKey
}
Then compute:

enum SecurityLevel: Int {
  case weak
  case moderate
  case strong
  case hardened
}
This becomes a derived property on Account or AccessProfile.

2) Token / Credential Lifecycle (dynamic decay)
This is how access degrades over time.

Different from security level.

Examples:

OAuth access token → expires in hours

refresh token → expires in weeks/months

API key → “never expires” but should rotate

session cookies → short-lived

support PIN → effectively static but weak

So you need a lifecycle model, not just a date.

struct CredentialLifecycle {
  var issuedAt: Date?
  var expiresAt: Date?
  var rotationInterval: TimeInterval?
  var lastRotatedAt: Date?
  var autoRefresh: Bool
}
Why these must be separate
This is a critical modeling decision.

Option A — Separate (correct)
Security Level = quality
Lifecycle = time decay

This lets you express:

Strong but expiring soon

Weak but permanent

Strong and recently rotated

Weak and stale (worst case)

Option B — Combine (bad)
People try to collapse this into one “status” field.

You lose:

clarity

prioritization

ability to compute “Needs Attention”

What this unlocks (important)
Now you can compute risk + urgency.

Example:

Account: AWS

Security: strong (MFA + scoped keys)

Token: expires in 2 days

→ high urgency, low structural risk

Versus:

Account: legacy bank login

Security: weak (password only)

Token: never expires

→ low urgency, high structural risk

Those should surface differently.

UI implication (this is where it gets good)
Each wallet item (account) should show:

Security Level badge

Expiration / decay indicator

Something like:

Schwab Brokerage

import Foundation

public struct VaultItemID: Hashable, Codable, Sendable {
  public let rawValue: UUID

  public init(rawValue: UUID = UUID()) {
    self.rawValue = rawValue
  }
}

public enum VaultItemKind: String, Codable, Sendable {
  case identity
  case password
  case passkeyReference
  case recoveryCodes
  case supportPIN
  case apiKey
  case oauthAccessToken
  case oauthRefreshToken
  case sessionToken
  case certificateReference
  case sshKeyReference
  case authorizationGrant
}

public enum VaultSensitivity: String, Codable, Sendable {
  case standard
  case high
  case enclaveBacked
}

public struct VaultItem: Identifiable, Codable, Sendable {
  public let id: VaultItemID
  public var kind: VaultItemKind
  public var title: String
  public var accountID: UUID?
  public var identityID: UUID?
  public var serviceID: UUID?

  public var sensitivity: VaultSensitivity
  public var metadata: VaultItemMetadata
  public var lifecycle: VaultLifecycle?
  public var secretReferenceID: UUID?

  public init(
    id: VaultItemID = .init(),
    kind: VaultItemKind,
    title: String,
    accountID: UUID? = nil,
    identityID: UUID? = nil,
    serviceID: UUID? = nil,
    sensitivity: VaultSensitivity,
    metadata: VaultItemMetadata,
    lifecycle: VaultLifecycle? = nil,
    secretReferenceID: UUID? = nil
  ) {
    self.id = id
    self.kind = kind
    self.title = title
    self.accountID = accountID
    self.identityID = identityID
    self.serviceID = serviceID
    self.sensitivity = sensitivity
    self.metadata = metadata
    self.lifecycle = lifecycle
    self.secretReferenceID = secretReferenceID
  }
}

public struct VaultItemMetadata: Codable, Sendable {
  public var issuer: String?
  public var username: String?
  public var maskedValueHint: String?
  public var scopes: [String]
  public var lastVerifiedAt: Date?
  public var lastUsedAt: Date?
  public var tags: [String]

  public init(
    issuer: String? = nil,
    username: String? = nil,
    maskedValueHint: String? = nil,
    scopes: [String] = [],
    lastVerifiedAt: Date? = nil,
    lastUsedAt: Date? = nil,
    tags: [String] = []
  ) {
    self.issuer = issuer
    self.username = username
    self.maskedValueHint = maskedValueHint
    self.scopes = scopes
    self.lastVerifiedAt = lastVerifiedAt
    self.lastUsedAt = lastUsedAt
    self.tags = tags

Can you have encrypted sandbox storage in an app too?

I’m checking the current Apple-side primitives, because the right answer depends on whether you mean “encrypted by the system,” “encrypted with my own app key,” or “Secure Enclave-backed.”🔐

Thought for 12s
Yes. You can have encrypted storage inside the app sandbox, but there are three different patterns and they are not equivalent.

The first is sandboxed file storage with Apple’s file/data protection. App Sandbox restricts access to your app’s container, and Apple documents container protection for local app data. On Apple platforms you can also use data protection for files so the file contents are protected based on device lock state, and on iOS Apple explicitly documents encrypting an app’s files with Data Protection. On macOS, app and app-group containers are also part of Apple’s local-data protection story. 

The second is Keychain storage. That is for small secrets and key material, not your whole database. Apple’s Keychain Services docs describe it as an encrypted database for small bits of sensitive user data, and the accessibility class controls when a keychain item is available, for example only when unlocked. Apple also documents stronger accessibility classes like kSecAttrAccessibleWhenPasscodeSetThisDeviceOnly, which ties availability to the device being unlocked and having a passcode set. 

The third is app-managed encryption. This is the one you probably want for Vault. In that design, you store your larger blobs or SQLite database in the sandbox, but encrypt the payload yourself. The wrapping key or private key lives in Keychain, and if you want hardware-backed protection, Apple documents protecting keys with the Secure Enclave. Also, keys bound to the Secure Enclave are not exportable in the normal “store key as data” sense, which is exactly the property you want for a custody layer. 

So the practical answer is:

For Vault metadata and bulk state, yes, use sandbox storage.
For raw secrets, use Keychain.
For high-value encrypted files or DB rows, encrypt them yourself with a key protected by Keychain or Secure Enclave, then store the ciphertext in the sandbox. 

The clean architecture for your app is:

Vault app:
stores accounts, identities, deadlines, notes, indexes, and encrypted blobs in the sandbox.

Keychain daemon or secure service:
stores secret material, wrapping keys, and access-control policy.

Secure Enclave:
protects private keys or key-unwrapping operations, not your entire sandbox directly. Apple positions the Secure Enclave as a hardware-based key manager, which is the right mental model. 

The wrong branch is “I’ll just put everything in UserDefaults/Core Data/SQLite inside the sandbox and call it secure because it’s sandboxed.” Sandboxing limits who can reach the files; it does not replace cryptographic protection for your own high-value payloads. The other wrong branch is “I’ll put my whole database in Keychain.” Keychain is the wrong tool for bulk storage and app state. Apple’s docs point it at small sensitive items. 

For your product, I’d use this split:

SQLite in the sandbox for normalized objects.
Encrypted attachment store in the sandbox for larger secret-bearing payloads.
Keychain for secret refs, wrapping keys, and small credentials.
Secure Enclave for private keys or unwrapping/signing gates.
App Group container only if multiple processes need shared access. Apple provides the group-container URL API for that. 

A good first-pass rule set is:

Use file protection for all local files you can.
Use Keychain accessibility like WhenUnlocked or stronger for credentials.
Use ThisDeviceOnly variants for secrets you do not want synced or migrated broadly.
Envelope-encrypt your SQLite blobs or sensitive records if Vault is meant to feel materially stronger than a normal app. Apple documents both keychain accessibility controls and Secure Enclave-backed keys for that foundation. 

So yes: encrypted sandbox storage is absolutely a valid design. The sharp version is not “sandbox or keychain.” It is “sandbox plus app-layer crypto, with keys anchored in Keychain/Secure Enclave.”

That means there are at least two decision trees.

The first tree is about protection model.

Option A is the weak model: encrypt the artifact with a password or device-local key and store it like a file. That is fine for ordinary storage, but it is not the interesting thing. It can be copied, migrated, brute-forced depending on implementation, and it does not really create the feeling of “this agent is mine.”

Option B is the strong model: compile the agent artifact, then wrap its opening key to a device-bound or identity-bound key, ideally with Secure Enclave-backed unwrapping on Apple platforms. That means the artifact can exist as a file, but only the intended owner’s trust chain can open it. That is the real magic. The file becomes portable-ish, but not intelligible without the right cryptographic identity.

The second tree is about execution model.

Option A is “open and inspect.” The vault decrypts the whole artifact and hands the runtime the plain contents. Easier to build, worse security posture.

Option B is “execute through controlled revelation.” The runtime only reveals the minimum needed parts: maybe policy first, then memory shards on demand, then tools under attestation or biometric unlock. That is much better because the agent becomes more like a sealed instrument than a ZIP file.

That second option is the right one.

a trading agent capsule

a tax prep agent

a family office ops agent

a legal entity agent

a broker integration agent

In fact, once you introduce sealed agents, identities, and different trust boundaries, a single-vault-per-user model becomes the wrong default.

The better question is not “can a user have multiple vaults,” but:

what boundary does a vault represent?

That gives you the real design decision tree.

A vault can represent one of several boundaries:

A personal boundary
This is “my private vault.” Personal credentials, private notes, private agents, personal tokens.

An entity boundary
One vault for an LLC, fund, trust, or family office. Different ownership, different documents, different policies.

A functional boundary
One vault for trading, one for infra, one for household ops, one for travel, one for health admin. This is less strong than entity separation, but still useful.

A security boundary
One vault with biometric-only reveal and stronger enclave policies, another with lighter access for ordinary accounts.

An sharing boundary
One vault shared with a spouse, business partner, or team. Another kept strictly private.

That is why multiple vaults are not just allowed — they are structurally important.

The bad option is single global vault with categories inside it. That sounds simpler, but it mixes trust domains. Then you end up inventing complicated per-item permissions, reveal rules, and sharing semantics inside one giant container. It becomes messy fast.

The better option is:

User
owns or has access to
many Vaults
and each Vault contains
many Vault Items / Accounts / Agents

So the hierarchy is:

User
  → Vault
    → Account / Identity / Token / Agent / Document
And the important twist is that vault membership is not the same as ownership.

A user can:

own a vault

co-manage a vault

only have operator rights

only view metadata

be blocked from revealing secrets

That is a much stronger model.
