Let’s tighten it into something structurally useful.

1. Reframe: KOMA = Execution Cell, Not Agent
Drop the “agent” framing entirely.

A KOMA should be:

A deterministic execution unit

With bounded state

And a well-defined contract (I/O + side effects)

Think less Fuchikoma, more:

Erlang process

Unix process

Swift actor (but stricter)

A KOMA is closer to a micro–work node in a graph.

2. The Minimum Viable KOMA (MVP Contract)
If it can’t be expressed this simply, it’s not a KOMA:

KOMA {
  Input
  State
  Transform
  Output
}
More concretely:

Input: typed payload (market data, text, events)

State: minimal, preferably serializable (SQLite row, struct blob)

Transform: pure or near-pure function (can call LLM, but isolated)

Output: event(s), mutation(s), or downstream payload

If it leaks global context → reject it.

3. Composition: Work Graph > Knowledge Graph
You already identified this correctly.

KOMAs should compose into a Work Graph, not a knowledge system.

Example (Tau trading system):

[Market Data KOMA]
      ↓
[Feature Extraction KOMA]
      ↓
[Vol Surface KOMA]
      ↓
[Strategy Selection KOMA]
      ↓
[Execution KOMA]
      ↓
[Journal KOMA]
Each node:

Stateless or lightly stateful

Replaceable

Parallelizable

This is where your “cellular automata” instinct is correct.

4. Two Competing Architectures (Decision Tree)
Option A — “Fat Agent KOMA” (❌ Reject)
KOMA = mini assistant with:

memory

reasoning

tools

personality

Why this fails:

Entangled state → impossible to debug

Non-deterministic outputs → bad for trading + infra

No composability → you get monolith agents

This is where most “AI agent frameworks” die.

Option B — “Thin Functional KOMA” (✅ Adopt)
KOMA = constrained execution unit:

Single responsibility

Typed I/O

Explicit dependencies

No hidden memory

Benefits:

Deterministic pipelines

Swap implementations (LLM ↔ heuristic ↔ SIMD)

Easy to run in:

Firecracker

Your minimal ASM boot env

This aligns with your “compile CLI + drop into environment” model.

What you are actually after is not a character unit. It is a logic-bearing mobile cell. A thing that can carry a local rule, move through a work graph, and participate in larger behavior without needing a giant central brain. 🤖

So I would redraw the model like this:

A KOMA is the shell.
A Logikoma is the shell plus executable logic contract.

That distinction matters.

A plain KOMA can just be a container or chassis:

identity

transport

state envelope

capabilities

routing metadata

A Logikoma is when that chassis contains:

local decision rules

transformation behavior

bounded memory

side-effect permissions

composition protocol

That makes it a real primitive.

The failure mode before was that KOMA risked becoming aesthetic taxonomy. A naming system for little agents. That is weak. A Logikoma is stronger because it implies that the unit is valuable even when stripped of personality, branding, and presentation.

The right mental model is not “mini assistant.”
It is closer to:

a rewrite rule with legs

a mobile predicate

a stateful combinator

a dispatchable reasoning cell

That gives you something you can actually build systems from.

The cleanest split is probably this:

1. KOMA = body
The body holds transport, lifecycle, permissions, serialization, scheduling.

2. LOGIC = rule
The rule defines what gets inspected, transformed, emitted, or requested.

3. LOGIKOMA = body + rule
Now you have a unit that can move through the system and do work.

That means composition becomes much more rigorous.

Instead of saying:
“these agents collaborate”

you say:
“these Logikomas compose through explicit contracts”

For example in Tau:

QuoteIngestLogikoma

SurfaceBuildLogikoma
  → Builds IV / skew / term structure

RegimeClassifyLogikoma
  → Tags market state

JournalEmitLogikoma
  → Writes trace + rationale
That is already much better, because each one has a legible logical role.

The real gain is that a Logikoma can be described as a very small algebra:

Logikoma<I, S, O> {
  observe: (I, S) -> Observation
  decide: (Observation, S) -> Decision
  act: (Decision, S) -> (O, S, Effects)
}
That is the core.

Or more aggressively:

L = inspect ∘ select ∘ transform ∘ emit
Now you can reason about them mathematically, test them, replay them, swap them, and compile them into different runtimes.

That is where the idea stops being gimmick and starts becoming infrastructure.

Two paths emerge here.

Path A: anthropomorphic agent objects.
You give each unit memory, style, tools, freeform planning, and broad permissions.

This is seductive, but bad.
Why it is bad:

hidden state

hard to replay

impossible to optimize

no sharp execution boundary

turns everything into “LLM soup”

Path B: sharply bounded Logikomas.
Each unit has one local logic domain, explicit I/O, controlled effects, and replaceable internals.

This is the right path.
Why it wins:

deterministic enough for trading and infra

composable

schedulable

portable across Swift, C, ASM, WASM

lets you substitute heuristics, local models, or remote models later

So the decision tree is straightforward.

If your goal is expressive demos, anthropomorphic KOMA works for a while.
If your goal is a real execution substrate for Tau, CLIA, and future work graphs, Logikoma is the only serious option.

The other important refinement is this:

A Logikoma should not “know everything.”
It should only know:

its current input

its own bounded state

a small capability surface

maybe a local policy table

Anything else should be pulled in explicitly.

That means no hidden omniscient context window by default. That is how you preserve system shape.

In Swift, I would start here:

public protocol Logikoma: Sendable {
  associatedtype Input: Sendable
  associatedtype State: Sendable
  associatedtype Output: Sendable
  
  func observe(_ input: Input, state: State) async throws -> Observation
  func decide(_ observation: Observation, state: State) async throws -> Decision
  func act(
    _ decision: Decision,
    state: State
  ) async throws -> Transition<Output, State>
  
  associatedtype Observation: Sendable
  associatedtype Decision: Sendable
}
And the transition object:

public struct Transition<Output: Sendable, State: Sendable>: Sendable {
  public let output: Output
  public let newState: State
  public let effects: [Effect]
  
  public init(
    output: Output,
    newState: State,
    effects: [Effect] = []
  ) {
    self.output = output
    self.newState = newState
    self.effects = effects
  }
}
Now you have an honest machine.

Notably, this design lets you do something powerful:
you can keep the outer KOMA runtime fixed, and vary only the logic style.

For the same Logikoma slot, you can install:

pure deterministic rules

statistical classifier

local Apple model

OpenAI model

hand-tuned trading heuristic

That is the right abstraction boundary.

So I would define the stack like this:

KOMA runtime: lifecycle, queueing, routing, persistence, permissions

Logikoma contract: observe / decide / act

Effect system: the only place side effects happen

Work graph: composition of Logikomas

Knowledge graph: optional backing store, not execution substrate

That last part matters a lot.

Knowledge graph is reference.
Work graph is motion.
Logikoma belongs to motion.

That is why the name works better. It implies locomotion plus logic. A unit that carries operational thought through a system.

if KOMA only means “a cute named agent class,” it dies.
If Logikoma means “the smallest mobile unit of executable reasoning,” then you have a foundation.

You want the second one. The first one is basically how you get “agent mush.”

So I would now upgrade the definition:

A Logikoma should satisfy at least five invariants.

First, bounded responsibility. It should own one logical pass, not a whole workflow.

Second, typed edges. Its upstream and downstream contracts should be legible.

Third, replayability. You should be able to rerun it with the same input and inspect deltas.

Fourth, diagnostic output. It should say what it observed and why it transformed.

Fifth, composability. It should fit inside a scheduler, queue, or streaming chain.

That last one is the Maps lesson. If it cannot survive being slotted into a larger pipeline, it is not a real production primitive.

public struct PipelineStage<I: Sendable, O: Sendable>: Sendable {
  public let name: String
  public let run: @Sendable (I) async throws -> O

  public init(
    name: String,
    run: @escaping @Sendable (I) async throws -> O
  ) {
    self.name = name
    self.run = run
  }
}
And then a pipeline runner:

public struct LogikomaPipeline<Input: Sendable, Output: Sendable>: Sendable {
  private let runBlock: @Sendable (Input) async throws -> Output

  public init(
    _ runBlock: @escaping @Sendable (Input) async throws -> Output
  ) {
    self.runBlock = runBlock
  }

  public func run(_ input: Input) async throws -> Output {
    try await runBlock(input)
  }

  public func then<NextOutput: Sendable>(
    _ next: @escaping @Sendable (Output) async throws -> NextOutput
  ) -> LogikomaPipeline<Input, NextOutput> {
    LogikomaPipeline<Input, NextOutput> { input in
      let intermediate = try await self.run(input)
      return try await next(intermediate)
    }
  }
}
