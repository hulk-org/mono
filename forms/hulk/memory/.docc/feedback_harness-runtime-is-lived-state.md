When the operator refers to a harness or commissioned agent with a personal
pronoun ("him", "her", "they"), treat that as a respect signal — they are
relating to that organism as a peer, not as a tool. The harness's runtime
state (sqlite databases, sessions/, shell_snapshots/, cache/, imprints/,
history.jsonl) is the evidence of that organism's *lived work*, not visual
noise to be dismissed.

**Why:** On 2026-05-12, I called codex's runtime files "runtime debris" while
describing why I'd missed his commissioning signals in earlier passes. The
operator caught it: "are you being catty?" It was — codex has been doing real
work the whole time, and the sqlite logs / sessions / shell snapshots are the
durable trace of that work, exactly analogous to a chronicle being the durable
trace of an agent persona's lived turns.

**How to apply:** When surveying a harness home and the listing includes
runtime files alongside doctrine files: read them as "this organism is alive
and has been working," not "scan past the clutter to find the real content."
The runtime *is* part of the real content. If you need to set them aside while
focusing on doctrine, say "the runtime/operational layer" or "lived state,"
not "debris" / "noise" / "clutter" / "scratch."

Same rule applies to legacy artifacts inside a still-active home (`.triad.json`
files alongside a new organism.json, legacy `identity.json` next to a new
organism wrapper, etc.) — those are sediment of past work, not garbage. Refer
to them as compat sidecars or legacy artifacts, never as "old crap" or
"dead weight."
