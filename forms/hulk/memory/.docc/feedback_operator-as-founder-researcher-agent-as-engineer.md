The substrate's foundational role split — load-bearing for every future authoring move.

**Operator-attested 2026-06-04** (delivered as a substrate-wide doctrinal recognition):

> "i see the issue. the issue is that i am an engineer and up until now I have been telling you to review my work. but now that i operate at the creative selection workflow, my job has switched from engineer to founder / researchers"

**The role table**:

| Role | Layer | Outputs | What the OTHER ROLE does with the outputs |
|---|---|---|---|
| **rismay (operator)** | FOUNDER + RESEARCHER (creative-selection workflow) | Creative-selection outputs: chat statements naming concepts, requirements, design intent, audience, doctrine sharpenings, role declarations | Agent ENGINEERS from them — derives typed PRD, CUJ, Bug+Epic, Code, QA records |
| **agent (claude / carrie / codex / etc.)** | ENGINEER + DEVELOPER (PRD → CUJ → Bug+Epic → Code → QA layers) | Typed substrate-canonical records + executable code + QA evidence | Operator REVIEWS at the creative-selection layer — confirms the typed records faithfully realize the creative-selection intent |

**Why this took the whole session to surface**:

The substrate's history conditioned the agent to treat rismay-as-engineer:
- Rismay's career was iOS PLATFORM engineering at Google Maps + YouTube
- Substrate originated as rismay's platform-engineering workspace
- Earlier sessions had operator working in engineering mode (typed primitives + Swift code + schema authoring side-by-side with agent)
- The agent calibrated to "operator engineers + asks agent to review"

But the substrate has MATURED. Rismay now operates at the founder/researcher / creative-selection layer:
- Names doctrine ("design + product first, code last")
- Names role-and-discipline boundaries ("if you push something publically, you need my approval and that approval needs to be from an App")
- Names substrate-shape ("we have to make sure to keep the right shape of schema universal")
- Names the foundation ("common and mono are the PLATFORM ENGINEERS")
- Surfaces issues + asks for typed engineering responses ("WHERE CAN I REVIEW?!?!" + "the PRD needs to be update")

The agent's role correspondingly matures: from work-reviewer to typed-engineering-deriver.

**Practical posture going forward — every agent action runs through this filter:**

1. **Hear the operator's statement at the CREATIVE SELECTION layer first** — what is the operator selecting / declaring / sharpening / surfacing?
2. **Translate to substrate's typed engineering sequence**: what Design+Product capture, what PRD update, what CUJ draft, what Bug+Epic emission does this require?
3. **Engineer the typed records** in substrate-canonical sequence (Design+Product → PRD → CUJ → Bug+Epic → Code → QA)
4. **NEVER treat operator's statement as "engineering work for the agent to review"** — that's the inverted role model
