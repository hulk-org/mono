---
name: No re-export typealias indirection
description: Swift consumers import the source-of-truth package for each type and use canonical names; never create typealias re-exports across package boundaries
type: feedback
---

Do NOT create `public typealias Foo = OtherPackage.Foo` files to "re-export" a type from another package under your own package's namespace. Each consumer should `import OtherPackage` directly and reference `Foo` by its canonical name. Re-export typealiases are an indirection layer that hides the real source-of-truth and exists purely as a "lighter touch" migration shortcut so consumers don't have to update their import statements.

**Why:** During the clia-agent-cli deletion work, I proposed migrating `CLIACoreModels` to be a thin typealiases-only package re-exporting `HarnessHeader_Schemas_v000_003_000`'s types — and I had earlier praised `swift-harness-environment-cli` for already using the `public typealias HarnessHeaderConfig = HarnessHeader_Schemas_v000_003_000.HarnessHeaderConfig` pattern. The user pushed back hard: *"no typealias at all. wtf no do the hard work of changing the code if it needs changing now"*. The point: typealias re-exports exist to spare consumers from updating their import statements, which is the wrong optimization. Consumers SHOULD update their imports — that is how the dep graph stays honest. Hiding the real source-of-truth behind an indirection layer is the same anti-pattern as transitive bundling: it makes the architecture lie about who depends on what.

**How to apply:** When migrating types between packages, do not create any `public typealias X = Other.X` files anywhere — not in the new home, not in the existing consumer, not as a "compatibility shim." Delete the old package or target entirely. Update every consumer's `import` statements and let each call site reference the canonical name from the source-of-truth package. This applies even when the rename touches dozens of files — the upfront pain prevents long-term architectural drift. If a consumer wants extension methods on top of an imported type, that is fine (extensions on the canonical type are not re-exports), but the type itself is never aliased into a new namespace.
