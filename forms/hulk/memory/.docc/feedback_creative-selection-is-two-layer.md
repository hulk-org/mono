## The split

**TOP layer — operator-generative:**
- Sets possibility-space (which typed options exist for consideration)
- States typed principles (which constraints those options must satisfy)
- Interprets the substrate's typed verdict + audit + analyses
- Decides next move (accept / narrow / relax / pin / snapshot / iterate)

**BOTTOM layer — substrate-determinative:**
- Applies the operator's typed principles to the operator's possibility-space
- Mechanical reduction; no operator agency at this layer
- Produces typed verdict variant (`.singleton` / `.multi` / `.empty`) + typed audit trail
- Surfaces typed analyses (minimum cover, cofactor, sensitivity, implication, clustering, contraction, etc.)

**Agency split axis:** operator owns inputs (top) + interpretation (top). Substrate owns mechanical reduction (bottom). Neither layer can do the other's job — operator-only collapses to vibes; substrate-only collapses to rigid automation.

## How to apply

When authoring substrate tools for design conversations:

1. Test every UI surface against the two-layer question: "is this move OPERATOR AGENCY or SUBSTRATE ENFORCEMENT?" Ambiguous answers indicate a missing typed record.
2. Visually + semantically separate the operator-agency surfaces (cascade picker, add-constraint, add-option, snapshot, provenance) from the substrate-mechanical surfaces (verdict pill, audit trail, all analyses).
3. NEVER use the language "the substrate picked X for me" — the substrate did NOT pick. The operator picked the inputs; the substrate showed the typed mechanical result. Preserve naming discipline.
4. Verdict must be typed; audit must be typed and inspectable; analyses must be typed projections of the audit. The substrate must NEVER hide the mechanical reduction behind a "recommendation" surface.
5. Spawn-software and maintain-software (the other two meta-cycle phases) will each instantiate this same two-layer typology. The split is workflow-shaped, not application-specific.
