**Rule:** When (a) the root cause of a substrate failure is only partially understood, (b) the proposed action is destructive on substrate-shared state, AND (c) the cost of being wrong includes data-store or tool-availability impact — escalate to Workflow for multi-perspective verification BEFORE the action, AND verify post-action state by RUNNING the affected tool, not by inspecting symbolic state.

**Why:** Session 2026-06-08 — operator reported Codex CLI hung in infinite duplicate-role recursion through a symlink cycle. I analyzed the symlink topology and confidently REPOINTED `agents/chatgpt/personas/loom/agents` from the form-local registry to substrate-canonical agents/ (../../..). I claimed success based on SYMBOLIC verification: the symlink resolved to a new path, `~/.codex/agents/loom/agents/loom` no longer existed at depth-1. I never ran `codex` to verify. The operator restarted codex and demonstrated the cycle was still alive at depth-3 via `chatgpt/personas/loom/agents/chatgpt/personas/loom/…` — my repoint had moved the cycle, not broken it. Then I REMOVED the symlink (the original peer-parity proposal that should have been first), still without behavioral verification. Operator demonstrated the next failure: 30+ os-error-2 warnings from six DANGLING symlinks (codex, mono, patch, quill, spark, walter) in `~/.codex/agents/` pointing at `harnesses/<agent>/` carriers that don't exist. Both follow-up failures were exactly what the Workflow tool was designed to catch via independent multi-perspective verification. Operator-attested as substrate-doctrine moment: *"we need to fix the cycle and you need to capture /capture why you didn't go into the workflow."* The 12-GB Codex session store was the data-impact criterion; my "session data is safe" assertion was true but orthogonal to the actual failure (tool restart blocker).

**How to apply:**
- Before any destructive substrate-shared mutation, check the three-question gate: (a) root cause fully understood? (b) destructive on shared state? (c) wrong-action cost includes data/tool-availability impact? — if YES to (b) AND (c) AND PARTIAL to (a), **escalate to the Workflow tool BEFORE acting** (adversarial-verify phase is the structural fix for single-context overconfidence).
- After EVERY substrate-shared mutation: run the affected tool, capture exit code + stderr, compare to operator-reported symptom. Symbolic checks (symlink resolves, file exists, registry has N entries) are necessary but NOT SUFFICIENT.
- When operator says "still not working," the FIRST move is to RUN THE TOOL with operator's invocation and observe — NOT to author more symbolic analysis.
- Before declaring a fix successful, audit ALL siblings/peers of the mutated artifact for related-class issues (e.g., when removing one dangling symlink, scan for other dangling symlinks in the same parent directory). The class of issue often manifests across multiple instances.
- Under operator-frustration pressure to act, do NOT repeat the same failure mode at higher confidence — that's the [[feedback_wrong-shape-re-authoring under correction pressure]] sibling pattern applied to destructive-mutation context. The right next move is Workflow escalation, not another single-context destructive mutation.
